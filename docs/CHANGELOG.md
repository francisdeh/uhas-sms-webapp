## Railway dev-deploy prep: SMS allowlist + seed-data safety â€” âś… Done

First piece of prep work ahead of standing up a Railway dev environment (the [self-hosted Supabase template](https://railway.com/deploy/supabase-firebase-alternative)), which will run against the existing demo seed data with real email/SMS providers wired up rather than a mocked/local stack. That combination raised a real question: background jobs (leave decisions, fee reminders, promotion/announcement broadcasts) would fire against ~150 seeded students/guardians/staff â€” is that safe, and is it free?

- **Email is already safe, once one inconsistency is fixed.** Seed data mixes `@uhas.edu.gh` (the school's real domain) with `@example.com` (the guardian-generation loop already used this). `.example`/`.example.com` is IANA-reserved (RFC 2606) specifically so it can never resolve to a real inbox â€” standardized the bulk-generated "extra staff" rows (`apps/api/app/scripts/seed/identity.py`) onto the same safe domain the guardian loop already used. The 9 documented demo login accounts (`admin@uhas.edu.gh`, `teacher@uhas.edu.gh`, etc.) are deliberately untouched â€” those are real, intentional credentials the team signs in with, not bulk demo noise, and changing them would break documented onboarding instructions.
- **SMS needed a different fix than email, because it's billed per-message.** The seeded phone numbers (`+233200000XXX`-style) looked obviously fake but aren't â€” `020` is a real, currently-assigned Vodafone Ghana prefix, so these are syntactically valid numbers that could collide with a real subscriber. Checked whether Hubtel has a sandbox/test mode first (several of Hubtel's own doc domains, including the one cited in this codebase's `HubtelSmsProvider` docstring, weren't reachable to confirm either way) and whether Supabase's `GOTRUE_SMS_TEST_OTP` mechanism would cover this (it doesn't â€” that only intercepts Supabase Auth's own phone-OTP *login* flow; the app's own domain-notification SMS calls `HubtelSmsProvider` directly and never touches Supabase Auth at all).
- **New `AllowlistSmsProvider`** (`apps/api/app/integrations/sms/provider.py`) wraps whichever real provider is configured. Outside production, only phone numbers on a new `sms_dev_allowlist` setting (comma-separated) ever reach the real provider and get billed; every other number is logged and handed the same synthetic `sent` result `StubSmsProvider` already returns for the "not configured" case â€” no network call, no cost. This is the opposite shape from email's existing `email_dev_redirect` (which resends *everything* to one address, since SMTP send volume isn't metered the same way) â€” SMS is billed per-message, so the right behavior for dev/demo traffic is to suppress it, not reroute it. Wired into `get_sms_provider()` as an outer wrap, gated on `settings.env != "production"`, so production behavior is completely unaffected when the new setting is unset.

Backend: `ruff check`/`ruff format --check`/`mypy`/`alembic upgrade head`/`pytest` all clean (1032 passed, up from 1024 â€” 8 new tests covering the factory's Hubtelâ†’Arkeselâ†’stub resolution and the allowlist's allow/suppress behavior). No frontend changes. Deliberately scoped to prep work only â€” the actual Railway service provisioning, Inngest Cloud account setup, and org-repo migration are tracked as separate follow-up steps.

## Pre-deploy verification pass â€” report cards, account flows, UX gaps â€” âś… Done

With all 6 tiers of the 16-domain correctness audit closed, this PR is a fresh round of manual pre-deploy QA â€” clicking through report cards, account-creation flows, and modals rather than re-running the original automated audit â€” that turned up several real, previously-undetected bugs.

- **Report-card aggregate summed every graded subject instead of the student's best 6.** A real student's End-of-Term-2 aggregate was 69 when it should have been 43 â€” real BECE methodology only ever counts a candidate's 6 best grades (in practice their 4 core subjects plus best 2 electives), not every subject they happen to be graded in. `compute_aggregate` (`apps/api/app/features/exams/compute.py`) now sorts ascending and sums the first 6; the TypeScript mirror in `apps/web/src/features/exams/utils.ts` got the same fix â€” then, since duplicating this calculation was exactly the kind of divergence risk that caused the bug in the first place, the TS copy was removed entirely (it was only ever used as a fallback when `card.aggregate` was missing, which never happens on a real response) so the backend's value is the sole source of truth going forward.
- **Admin got a bare "page not found" opening a report card for a student with no currently-active class** (e.g. transferred out, withdrawn) â€” `ReportCardRepository.active_class_for` only ever matched an `Active` enrollment for the exam's academic year, so a withdrawn student's own historical report card 404'd even though the exam and scores still exist. Now falls back to the student's most recent enrollment of any status for that year, ordered so an `Active` one still wins if present. For the genuinely-never-enrolled case (zero enrollment rows for that year at all), the 404 now carries a distinct `no_enrollment` error code so the Admin report-card page can render an actionable in-app notice ("go to student profile" button) instead of Next.js's generic not-found page.
- **Every modal in the app couldn't scroll past the viewport's height.** Neither `apps/web/src/components/ui/dialog.tsx` nor `alert-dialog.tsx` had a `max-h`/`overflow-y-auto` on the popup â€” confirmed via a live check that the Edit Student modal's content (783px) genuinely overflowed its 468px-tall container with zero way to reach the Save button on a short viewport. This is a shared-primitive bug affecting all ~41 `DialogContent`/`AlertDialogContent` call sites in the app, not an isolated one; fixed once at the primitive level (`max-h-[calc(100vh-2rem)] overflow-y-auto`) rather than per-modal.
- **The "New account" modal's Staff/Guardian picker listed everyone, including people who already had a login** â€” clicking one of them would either 409 against their existing account or (worse) risk creating a second, divergent one for the same person. Now filters out anyone already appearing as a `linkedId` on an existing account. Picking a still-eligible person now also auto-fills their name/email/phone from their own record (previously you had to know and retype it yourself even though the record you just selected already had it) â€” Role/person selection was moved to the top of the form so it drives that autofill instead of silently overwriting fields already typed in below it.
- **Staff detail page had no way to actually provision a login.** `POST /staff/{staff_id}/login` (`UsersService.provision_staff_login` â€” infers role and sources email/phone straight from the staff row, no separate picker needed) has existed on the backend the whole time, but had zero frontend caller â€” the only working path was the generic Users page's "New account" flow, which makes you re-pick the role and re-find the same staff member you're already looking at. Added a "Create Login" button directly on the Staff detail page calling this existing endpoint.
- **A duplicate `uhas_id` produced a misleading error instead of a clear one.** `Staff.uhas_id` has a real DB-level `unique=True` constraint, but unlike email there was no explicit pre-check â€” `StaffService.create` relied on `insert_with_sequential_slug`'s generic retry-on-`IntegrityError` loop, which assumes every collision is a slug collision and retries with the next sequence number; a duplicate `uhas_id` doesn't change between retries, so it fails every attempt and surfaces as `"Could not generate a unique slug after retries."` â€” true but unhelpful. `StaffService.update` had it worse: no error handling around the flush at all, so the same collision would have been a raw unhandled 500. Both now explicitly check `StaffRepository.find_by_uhas_id` first and raise a clear `ConflictError("This UHAS Staff ID is already in use.")`.
- **Add Staff's phone field claimed something untrue.** Copy read "Used for this staff member's login (phone-OTP) once a login is provisioned" â€” but `LoginForm.tsx` explicitly rejects any non-Parent-role account that completes a phone-OTP flow (`"Phone sign-in is only available for parent accounts."`), so a staff phone number is never actually usable for login even though Supabase technically wires it up as OTP-capable. Corrected to describe what the number is actually for: SMS notifications (e.g. leave-request decisions) â€” staff sign in with email + password only.
- **Verified, not fixed**: investigated whether the guardian create-login/link-guardian flow (`GuardianTab.tsx`/`GuardianField.tsx`) had the same class of gaps as staff â€” it doesn't. It already shows a "Has login"/"No login" badge per guardian, only offers "Create login" when one doesn't exist, supports linking an existing guardian, creating a new one, or linking/creating one backed by a staff record (deduping via a `staffId` lookup so the same person never gets two guardian rows), and its "new guardian" copy already correctly states the phone/email entered is what the guardian will use to log in. No changes needed.
- Also fixed: missing icons on Student detail's Edit/Transfer Class buttons and Class detail's Change Class Teacher/Add Subject buttons (inconsistent with their sibling buttons, which all had one); a stale `suggestion.py` docstring claiming to mirror a TypeScript file that was deleted during the FastAPI migration (the frontend has never computed this locally â€” it just displays the API's `suggestedDecision`/`suggestedReason`).

Backend: `ruff check`/`ruff format --check`/`mypy`/`alembic upgrade head`/`pytest` all clean (1024 passed, up from 1020 â€” new tests cover the best-6 aggregate cap, the enrollment-fallback report-card fix, the `no_enrollment` error code, and both the create and update `uhas_id` duplicate-collision paths). Frontend: `lint`/`tsc --noEmit`/`vitest`/`build` all clean; no `api.d.ts` changes needed (no new routes/schemas â€” only internal logic, a new error code string, and UI changes). Manually verified every fix live in the browser: confirmed the real student's aggregate now reads 43; opened a withdrawn student's historical report card and confirmed it renders instead of 404ing; resized the viewport and confirmed the Edit Student modal's content genuinely overflows and scrolls; confirmed the New Account dropdown excludes all 8 staff who already have logins and correctly auto-fills on selection; created a real login via the new Staff "Create Login" button (confirmed the resulting account appeared on the Users page), then cleaned up the test-created Supabase Auth user and `users` row afterward.

## 16-domain correctness audit â€” Tier 3 atomic mutations â€” âś… Done

Fourth fix batch from the 16-domain correctness audit, closing Tier 3 â€” non-atomic multi-step mutations that were orchestrated as sequential independent API calls from the frontend instead of one backend transaction, so a failure partway through could leave a record in a broken half-state.

- **Student class transfer was withdraw-then-create as two separate calls.** `StudentDetail.tsx`'s transfer mutation issued a `PATCH` to withdraw the old enrollment followed by a `POST` to create the new one â€” if the second call failed (network blip, validation error), the student ended up with no active enrollment at all. Fixed with a new `EnrollmentsService.transfer_student` and `POST /enrollments/transfer` endpoint that does both inside one DB session/commit; the frontend mutation collapses to a single call. New tests cover the atomic swap, admin-only access, same-class rejection, and unknown-class validation.
- **Class-teacher reassignment was remove-then-assign as two separate calls.** `ClassDetail.tsx`'s "Change Class Teacher" dialog called `useRemoveClassTeacher` then `useAssignClassTeacher` â€” a failure between the two left the class with no primary teacher. Fixed with a new `ClassTeachersService.replace_primary` and `PUT /classes/{class_id}/teachers/primary` endpoint (handles delete-old + assign-new, or clearing to no teacher, in one transaction); the frontend collapses to one `useReplacePrimaryClassTeacher` call. New tests cover the atomic swap and clearing to null.
- **Editing a historical attendance session silently dropped students who'd since left the class.** Admin's "any past date" attendance editing is a deliberate feature (`AdminAttendancePicker`'s own copy says so), but `AttendanceService.upsert_session`'s roster-validation only ever checked *today's* active enrollment â€” so resubmitting an old session for a student since transferred out or deactivated rejected their record as "not enrolled," and since the backend replaces the whole session's records on every save, the student's attendance for that day was gone for good. Fixed by computing the existing session's previous records *before* roster validation and admitting `roster | previous_status.keys()` instead of `roster` alone â€” a student already on record for this specific session stays valid even after falling off today's roster. `AttendanceRecordRead` already joins the student's name/slug onto each record, so the frontend page needed no extra API call to keep showing them; it now merges any such student into the roster array from the session response it already has. New tests cover the preservation (withdraw a student mid-session, resubmit, confirm their record survives) and confirm a genuinely unrelated stray ID is still rejected.

Backend: `ruff check`/`ruff format --check`/`mypy`/`alembic upgrade head`/`pytest` all clean (1002 passed, up from 991 â€” 10 new tests: 4 for the transfer endpoint, 4 for the class-teacher endpoint, 2 for the attendance preservation fix). Frontend: `lint`/`tsc --noEmit`/`vitest`/`build` all clean; `api.d.ts` confirmed in sync via a fresh local uvicorn instance (the repo's own drift-check script collides with an unrelated process on port 8001). Manually verified all three flows end-to-end in the browser: swapped JHS 1's class teacher and swapped it back, transferred a student between classes and back, and re-saved a live attendance session confirming the everyday save path still works unchanged â€” all three left no stray state behind.

Remaining tiers from the same audit (audit-log gaps, UX/data-quality polish) are tracked for follow-up PRs.

## 16-domain correctness audit â€” Tier 2 account/session safety â€” âś… Done

Third fix batch from the 16-domain correctness audit, closing Tier 2 â€” account and session safety, the cluster most relevant once real staff accounts exist in production.

- **Deactivating a staff member never actually revoked their login.** `StaffService.set_active` only flipped `staff.is_active` â€” the real enforcement mechanism (`users.is_active` + a Supabase ban) was never touched, so a "deactivated" Teacher, DeputyHead, or Admin kept full access indefinitely, and the deactivate-confirmation dialog's own copy ("They will lose system access") was simply false. Fixed by having `StaffService.set_active` look up the linked `users` row (via the same `find_user_for_linked` helper notifications already use) and call `UsersService.set_active` too, propagating the Supabase ban and audit log in the same transaction. `StaffTable.tsx`'s confirmation copy now matches `StaffDetail.tsx`'s (both were subtly different before).
- **Changing a staff member's role never touched what actually gates access.** `StaffService.change_role` only ever updated `staff.system_role` â€” never `users.role` or the JWT's `app_metadata.role`, which is what every `RequireAdmin`/role dependency actually reads. An admin "promoting" someone got a positive `"Role updated"` toast while the person's real dashboard access, notification routing, and leave-approval eligibility stayed on their old role â€” the two fields could diverge forever with no way to reconcile them short of a direct DB edit. Fixed by syncing both in the same call, resending the full `app_metadata` object (`role`/`school_id`/`linked_id`) since Supabase's admin API replaces rather than merges it.
- **Nothing stopped an Admin from deactivating their own account, or the school's last active Admin.** Added a guard to `UsersService.set_active`: an Admin-role deactivation is refused if the target is the actor themself, or if it would leave zero other active Admins in the school. Scoped to Admin-role targets only â€” the existing self-service `/me/deactivate` path (for Teacher/DeputyHead/Accountant leaving on their own terms) is untouched and still works exactly as before; that path already independently blocks Admin self-deactivation unconditionally, so this is the admin-panel-initiated equivalent for the same protection.
- **Accountant was unreachable through the Staff UI.** Three separate places â€” the registration form's role picker, the Change Role dialog's role picker, and the Staff table's role filter pills â€” hardcoded `Admin`/`DeputyHead`/`Teacher` only, even though Accountant has existed as a fully modeled, styled role since Phase 5. All three now map over the shared `STAFF_SYSTEM_ROLES` constant instead of a hand-copied list, so a future role addition can't silently repeat the same miss. Also fixed the "division required" validation in both forms, which only checked `DeputyHead`/`Teacher` â€” Accountant would have hit the backend's real rule ("division required unless Admin") with no field shown to satisfy it; both forms now key off "any non-Admin role" instead.
- **2FA's step-up gate was described as "un-bypassable" in a comment that wasn't true.** Investigated the actual architecture: enrollment is fully opt-in with no persisted "has this account enrolled" signal anywhere (DB or JWT) â€” the Next.js proxy's AAL check is a live Supabase client-SDK call on every request, and FastAPI never reads the JWT's `aal`/`amr` claims at all, even though Supabase already puts them on the token. A client holding a still-valid `aal1` bearer token can call any backend endpoint directly. Building real API-layer enforcement needs a new persisted enrollment flag (written at enroll/unenroll/reset time, read by a new FastAPI dependency) that doesn't exist today â€” confirmed with the user this is out of scope for this PR; fixed only the misleading comment in `proxy.ts` to accurately describe the gate's real (UI-navigation-only) scope, and flagged the real fix as separate follow-up work.

Backend: `ruff check`/`ruff format --check`/`mypy`/`alembic upgrade head`/`pytest` all clean (994 passed, up from 991) â€” new tests cover the deactivation cascade (login revoked when linked, gracefully skipped when not), the role-change JWT sync (`users.role` + Supabase `app_metadata` both updated), and all three self/last-admin guard scenarios (own account rejected, last admin rejected, succeeds when another admin remains). Frontend: `lint`/`tsc --noEmit`/`vitest`/`build` all clean; `api.d.ts` confirmed still in sync (no schema changes this PR â€” only new FastAPI dependency params). Manually verified end-to-end in the browser: registered a real Accountant through the fixed Staff registration form (role + division fields both worked correctly, staff record created, then removed since it was a throwaway verification record); confirmed the Change Role dropdown lists all four roles too.

## 16-domain correctness audit â€” Tier 1 rollover fixes â€” âś… Done

Second fix batch from the 16-domain correctness audit, closing Tier 1 â€” the operational-blocker cluster around academic-year rollover and cross-division promotion.

- **Cross-division promotion target-class resolution was fundamentally broken.** `PromotionsRepository.next_year_classes_for_division` (and its 3 call sites in `promotions/service.py` plus the router's `_build_detail`) filtered next-year candidate classes by the *current* class's own division â€” so for any of the school's 3 cross-division boundaries (KG2â†’P1, P3â†’P4, P6â†’JHS1), the algorithmic auto-pick could never resolve a target, and the manual dropdown never even listed the correct next class. Every promotions fixture in the test suite used same-division JHS1-3 classes only, so this had zero test coverage before now. Fixed by switching all 4 call sites to the already-existing, school-wide `PromotionsRepository.classes_for_school_year` â€” `auto_pick_target_class` matches candidates by name, not division, so the full school-wide list is exactly what it (and the dropdown) need. The `submit_list` pre-flight's `division_has_next_year_classes` check (itself division-scoped, same bug) is replaced with a simpler and more correct "does the school have ANY next-year classes yet" check, since `prepare_next_year` always creates every division's classes together in one pass â€” there's no scenario where one division has next-year classes and another doesn't. Deleted the now-dead `division_has_next_year_classes`/`divisionHasNextYearClasses` helpers (and their tests) from both the Python and TS mirrors, since neither was called anywhere once the fix landed. New end-to-end regression test seeds a genuine Primary 6 â†’ JHS 1 scenario and asserts the auto-pick, the dropdown, and a real submit all now work.
- **`activate_next_year` had no completeness guard** â€” the exact gap that caused a live incident this session (activating the next year with only 2 of 11 classes promoted dropped 92 of 112 students' current enrollment app-wide). `SchoolsService.activate_next_year` now fetches every current-year class and every submission for the year, and refuses with a clear "N of M classes don't have an approved promotion yet: <names>" error unless 100% are approved. The Year Rollover card on Settings â†’ Calendar now fetches the same promotions overview the Admin overview page already uses, shows "X of Y classes promoted" (with the names of whichever aren't), disables Activate until every class is done, and gates the actual click behind a confirmation dialog â€” mirroring the existing bulk-approve confirmation pattern rather than a bespoke one.
- **Admin had no UI to approve or send back a promotion list** â€” the backend has always allowed it (`_assert_reviewer_can_review` short-circuits for Admin regardless of division, already covered by an existing backend test), but the Admin submission-detail page never rendered `ReviewFooter`, unlike the identical Deputy Head page. A division with no assigned Deputy Head had no way for anyone to move its submissions past `submitted`. ## Supabase Send SMS hook â€” relay phone-OTP through Hubtel â€” âś… Done

Second of two provider-wiring PRs, deliberately kept separate from the first (Brevo/Hubtel precedence swap) since this one adds a genuinely new piece â€” a public, unauthenticated webhook endpoint â€” rather than a simple factory-precedence change.

**The gap**: Supabase Auth's phone-OTP sign-in (the Parent login path) has always used a placeholder `[auth.sms.twilio]` block in `supabase/config.toml` with unset credentials, existing only to satisfy Supabase's "at least one provider must be enabled" requirement â€” a comment in that file even flagged this as "Phase 3 â€” not implemented here" months ago. Real OTP delivery never actually worked; only the pinned `test_otp` numbers did.

**The fix**: a new `POST /auth/send-sms-hook` endpoint (`app/features/auth/router.py`) registered as Supabase's custom **Send SMS hook** â€” Supabase now calls this instead of Twilio whenever it needs to deliver an OTP, and the endpoint relays the send through the app's own `get_sms_provider()` (Hubtel, per the previous PR). Every request is verified against the Standard Webhooks HMAC signature (new `standardwebhooks` dependency â€” the same open spec Supabase itself implements, not a Supabase-specific SDK) using a shared secret Supabase generates when you register the hook. Reads the **raw** request body before any JSON parsing, since signature verification must be over the exact bytes sent, not a re-serialized version.

**Deliberately fails closed**: unlike every other provider setting in this codebase ("missing config isn't an error, just skip"), a missing `SEND_SMS_HOOK_SECRET` here returns a 500 rather than silently accepting unsigned requests â€” this is the one place where "safe to skip" would actually mean "anyone who finds the URL can relay arbitrary SMS through the school's Hubtel account."

**Scope decision**: hook-triggered OTP sends do *not* get logged to `sms_log` like every other domain's SMS (fee reminders, leave notifications). Supabase's hook payload may not yet include our own custom `app_metadata.school_id` claim on a brand-new signup (it only reliably has Supabase's own `provider`/`providers` fields at that point), which would make `school_id` resolution unreliable for logging purposes â€” Hubtel's own dashboard already covers delivery/cost auditing for this flow, so the endpoint calls the provider directly rather than routing through `SmsService.send()`'s DB-logging wrapper.

**Testable locally, not just in production**: Supabase's local Auth container (running in Docker) reaches the FastAPI dev server via `http://host.docker.internal:8000/auth/send-sms-hook` on macOS â€” new `[auth.hook.send_sms]` block in `supabase/config.toml`. Real Hubtel OTP delivery can be exercised from a developer's own machine the same way production will.

Backend: `ruff check`/`ruff format --check`/`mypy`/`pytest` all clean (899 passed).
