# Changelog

Full development history for the UHAS SMS project, newest first. `README.md`'s Development Phases table is a slim index into this file — this is the canonical place for detailed "what shipped and why" narrative.

## Academic-year / term management deep-dive — ✅ Done

Closed the last backlog item from the pre-go-live gap audit's original six. A pre-design audit found four gaps: `apps/web/src/lib/academic-year.ts`'s hardcoded 4-entry `ACADEMIC_YEARS` array (enforced via `z.enum()` on class/exam creation and the year switcher) meant an Admin could no longer create anything for a new year — or even have the app recognize its own `school.academicYear` — once the real year advanced past the array's last entry, since `getCurrentAcademicYear()`'s fallback silently reverted to a hardcoded default; rolling to a new year was two disconnected manual steps (Promotions creates next-year enrolments, but never touches `schools.academic_year`, and an Admin had to separately hand-edit Settings with no validation); `schools.current_term` was purely cosmetic despite the original spec calling for a date-based auto-pick; and the parent dashboard's attendance-percentage calculation hardcoded the academic year as Sept 1–Aug 31 while report cards correctly resolved real `school_terms` dates for the same purpose.

Fixed all four as one pass:
- **Dynamic year source**: deleted the hardcoded array. Every year picker now derives its options from `school_terms` data plus the school's real current + next year (`next_academic_year()`, already shared with Promotions) — an Admin can always start preparing next year even before its `school_terms` rows exist. `getCurrentAcademicYear()` now trusts `school.academicYear` unconditionally instead of falling back to a stale default.
- **Rollover workflow**: two new Admin-only endpoints on the Settings → Calendar tab. **Prepare next year** copies the current year's classes forward (idempotent — skips ones that already exist) and pre-fills next year's `school_terms` by shifting this year's dates forward one year. **Activate next year** flips `schools.academic_year`, guarded to refuse while the current year's promotion season is still open, and audit-logged (`SCHOOL_YEAR_ACTIVATED`).
- **Term auto-pick with override**: new nullable `schools.current_term_override` column. A shared resolver (`app/features/schools/term_resolver.py`) computes the effective current term as override → whichever `school_terms` row's dates contain today → nearest term by date → the legacy stored value, in that order — never erroring even with incomplete data. The Calendar tab's term control is now "Auto — Term N" with an explicit pin.
- **Two downstream bug fixes**: the parent dashboard's attendance-percentage date range now resolves real `school_terms` dates instead of the Sept–Aug guess; exam creation defaults its `term` field to the resolved current term instead of always `1`.

Backend: new migration (`current_term_override`), `PrepareNextYearRead`/`SchoolUpdate` schema changes, 18 new pytest cases (6 pure resolver tests, 6 service-level rollover tests, plus existing suite updates) — all passing alongside the full existing suite. Frontend: `tsc`/lint/Vitest/build all clean; manually verified in-browser that Prepare creates real classes+terms, Activate correctly disables until preparation is done, the year switcher picks up the newly-created year with no code change, and a new exam's term field defaults to the resolved term. See `docs/superpowers/specs/2026-07-14-academic-year-term-management-design.md`.

Code review during this PR caught the attendance class-picker showing a raw class UUID instead of its name after selection — a Base UI `SelectValue` quirk (it doesn't automatically display the matching `SelectItem`'s children; it needs an explicit render-prop function, or the item's `value` and display text must already be identical) already fixed once on the Calendar tab's own new term-override selector earlier in this same PR. A full sweep for the same pattern across every `<Select>` in the app found **11 total instances**: the attendance class picker (`AdminAttendancePicker.tsx`), the "assign cover" substitute-staff picker (`LeaveRequestList.tsx`), fee-item and appointment class/teacher pickers (`FeeItemForm.tsx`, `ParentAppointmentsView.tsx`), the class-detail subject picker (`ClassDetail.tsx`), student registration's class picker (`StudentRegistrationForm.tsx`), both remaining audit-log filters (`AuditLogFilters.tsx`'s User and Target dropdowns), the exam-type picker (`ExamsManager.tsx`), the staff system-role picker (`StaffRegistrationForm.tsx`), the leave-type picker (`LeaveRequestForm.tsx`), and the fee-balances status filter (`BalancesTable.tsx`) — all fixed with the same render-prop pattern.

## Tier 4 follow-on bug sweep — ✅ Done

Six items the user flagged in one batch (staff email edit vs. Auth login, UUID leaks across the frontend, the Profile page's unclear "Documents" card, staff phone-login policy, broken self-service profile-photo upload, and a Settings-page button gap), each investigated independently via parallel research agents before any fix landed, so real bugs and non-bugs weren't conflated.

- **Staff email edit vs. Supabase Auth login** — investigated, found already safe: `StaffService.update` (and the equivalent `GuardiansService.update`) syncs an admin-driven email/phone change straight to Supabase Auth via the admin API (`update_user_by_id(..., email_confirm=True)`) in the same request, by deliberate design (admin edits are trusted, same as account creation) — structurally different from, but not buggy relative to, the self-service link-based confirmation flow. No code change needed.
- **Raw UUIDs leaking into the UI** — one shared root cause and several one-offs. `AutoBreadcrumb.tsx` had no way to resolve a dynamic route segment (e.g. a class/student/staff id) to its real name, so it silently fell back to rendering the raw id — fixed with a new `apps/web/src/features/shell/breadcrumb-context.tsx` (`BreadcrumbLabelProvider` + `useBreadcrumbLabel`) that lets any page feed the breadcrumb the entity's name once fetched; wired into all ~15 affected dynamic routes (attendance, classes, examinations, promotions, staff, students, report cards, assignments, class-reports, lesson plans, schemes, fee items). Also fixed: the attendance roster and the printable student ID card rendered `student.id` instead of `student.slug`; the parent dashboard's "My Children" list, the global search command palette's Students/Staff results, and the PSC report's "Staff ID" column all lacked a display-id field entirely and needed one plumbed through (the PSC fix required a small backend schema addition, `PscStaffEntry.slug`, plus a regenerated `api.d.ts`).
- **Profile page's "Documents" card** — not a bug, just unlabeled: it's a plain card appended to the bottom of the Profile tab (not a real tab), with no `CardDescription` and a generic "No documents uploaded yet." empty state, making it read as confusing dead UI even though the access-control and data plumbing all work correctly (an Admin's own profile does resolve via their own `staff` row). Fixed with real explanatory copy and a clearer empty state, differentiated by `canManage`.
- **Staff phone-login** — investigated and found to be a real, unintended gap: `LoginForm.tsx`'s `classify()` routes purely by input shape (phone-looking string → OTP flow), with no role check anywhere in the pipeline, so a staff account with a confirmed phone could complete phone-OTP login today, contradicting CLAUDE.md's documented "staff = email+password only, phone-OTP is Parent-only" design — and staff never got the onboarding SMS notice guardians do, since that path only fires when there's no email on file (staff always have one). Fixed by adding an explicit role check in `handleVerifyOtp`: phone-OTP now signs out and rejects any non-Parent role immediately after verification, with a clear error message. Code review caught a follow-on inconsistency: the Profile page's phone-field copy ("...before it becomes login-usable") was still shown identically to every role, contradicting the restriction just added — `PhoneChangeCard` gained a `usedForLogin` prop (`true` only for Parent) so staff see accurate copy ("used for contact and SMS notifications only — not for signing in") instead.
- **Self-service profile-photo upload** — two real, distinct bugs, found via a dedicated investigation plus a broader app-wide upload-mechanism audit (which confirmed every OTHER upload site — staff/student registration, StaffDetail/StudentDetail edits, Settings' branding logo — shares one working, unaffected code path). Bug A: an Admin/staff account provisioned without a linked `staff` row (possible via the generic "Add user" flow) silently lost the entire upload widget, since `canEditPhoto` requires a truthy `linkedId`. Bug B: `get-my-photo.ts` checked `linkedId.startsWith("STAFF-")`, but `linkedId` is a UUID, not a `STAFF-XXX` string — always false, so the sidebar/header avatar never showed a real photo for anyone, regardless of what was saved. Given both bugs and that `UserAvatar`'s initials fallback already renders everywhere reliably, removed self-service photo upload entirely (Profile page, Sidebar, Header) rather than fixing a two-bug feature nobody could actually see working — `apps/web/src/features/profile/queries/get-my-photo.ts` deleted, `userPhotoUrl` prop chain removed from `DashboardLayout`/`Sidebar`/`Header`, all four profile route pages simplified. Admin-set staff/student photos (registration forms, tables, printable ID cards) are untouched — a separate, working mechanism.
- **Settings page buttons** — confirmed via git history that `features/settings/` was never touched by the action-button-consistency pass (tier 3); all 6 tabs' Save buttons still used the pre-pass `variant="ink"`. Swapped to `variant="brand"` to match every other primary CTA in the app.

One bug found and fixed along the way, unrelated to the six flagged items: the new `breadcrumb-context.tsx`'s first draft caused an infinite render loop (`useBreadcrumbLabel`'s effect depended on the context value object itself, which was a new object identity every render since the Provider's `value={{...}}` wasn't memoized) — fixed by wrapping `setLabel`/`clearLabel` in `useCallback` and the context value in `useMemo`.

## Account emails: real provider + branded invite/reset/change — ✅ Done

Staff onboarding had no way to actually send a login invite — `StaffRegistrationForm.tsx`'s success dialog showed a hardcoded placeholder link. Fixed alongside a bigger gap the same investigation surfaced: `provision_login` (shared by staff and guardian logins) relied on Supabase's own generic invite email, off-brand and disconnected from the branded system built in slices 1–4; the same was true for password recovery and email-change confirmation, both triggered directly client-side (`supabase.auth.resetPasswordForEmail`/`updateUser({email})`) with zero backend involvement. All three now mint their Supabase link via the admin `generate_link` API (invite/recovery/`email_change_current`+`email_change_new`) and send it through our own Jinja2 templates instead of Supabase's mailer — verified against the local Supabase CLI stack, including that `generate_link` really does support secure dual-confirmation email-change and that a non-existent email really does raise a distinguishable `NotFoundError`. That last point is load-bearing for `POST /auth/reset-password` (new, public, IP-rate-limited): every branch — unknown email, cooldown active (new `users.last_password_reset_sent_at`, 5 min), Supabase itself reporting no such user — returns the identical 204 so the endpoint can't be used to enumerate accounts, mirroring the anti-enumeration guarantee `resetPasswordForEmail` already gave for free. New `POST /staff/{id}/login` mirrors the existing `POST /guardians/{id}/login` (infers role from the staff row's own `system_role`, no re-asking) and replaces the dead placeholder link with a real "Send invite" action. Production email provider is now Resend (`ResendEmailProvider`, takes precedence in `get_email_provider()`); local dev is Mailpit (`docker compose up mailpit`, view at `:8025`) — `SmtpEmailProvider`'s `user`/`password` became optional so the same class serves both an authenticated real SMTP server and Mailpit's no-auth catcher. See `docs/superpowers/specs/2026-07-13-account-emails-provider-design.md`.

## 6 item 8 slice 4 — Attendance absence notifications — ✅ Done

Fourth of the originally-scoped 4-PR initiative. `AttendanceService.upsert_session` is session-based — it deletes and re-inserts an entire class roster on every save, no per-student mark/correct method exists — so the core problem this PR solves is dedup: fetches the previous session's records into a `{student_id: status}` map before the delete, and only a genuine status *transition into* `"Absent"` notifies (a same-day resubmission that leaves an already-absent student unchanged stays silent; `Absent → Present → Absent` genuinely re-notifies). Recipient resolution follows the results-published/fee-reminder precedent: primary guardian only, batched — a guardian with two newly-absent children in the same session save gets one combined email + SMS, not two. `"Late"`/`"Excused"` stay silent; only `"Absent"` triggers. Single-direction (parent-facing only) — one preference pair, `{email,sms}_on_attendance_absent`. The one deliberate default flip in this initiative: `schools.notification_defaults.on_attendance_absent` defaults to `false`, not `true` like every other toggle — attendance is marked daily for potentially every student, a materially higher volume and more sensitive category than the occasional appointments/leave/results events, so a school opts in explicitly rather than this firing unannounced. See `docs/superpowers/specs/2026-07-12-attendance-absence-notifications-design.md`.

## Rate-limiting audit — ✅ Done
